# PRD — FleetOps Agent

**Owner:** Taha Hussain · **Status:** v1 (portfolio/demo) · **Date:** 2026-07-08

## Problem

Fleet maintenance managers answer dozens of ad-hoc questions a day — "which trucks are
overdue for service?", "what does fault code SPN-3226 mean and do we have the part in
stock?", "summarize last month's maintenance spend" — by hand-writing SQL or digging
through spreadsheets. This project demonstrates a **tool-calling AI agent** that answers
those questions over a fleet-maintenance database, with an **evaluation-first** workflow:
every capability is backed by a golden dataset, deterministic tool-selection checks, and
LLM-as-judge answer scoring wired into CI.

All data is **synthetic** (generated by `src/generate_data.py` with a fixed seed). Public
datasets and reference repos were used for schema inspiration only.

## Users & core flows

- **Fleet ops manager (demo persona):** asks natural-language questions in a dashboard,
  gets tables/answers with cited tool calls, downloads maintenance reports.
- **AI engineer (me / reviewer):** inspects eval results, re-runs the regression suite,
  extends the golden dataset.

## Success metrics

| Metric | Definition | Target |
|---|---|---|
| Tool-selection accuracy | % of golden questions where the agent's tool-call trajectory contains the expected tool(s), scored deterministically | ≥ 85% |
| Answer quality | Mean LLM-as-judge score on an explicit 1–5 rubric (correctness, groundedness, completeness) against reference answers | ≥ 4.0 / 5 |
| Guardrail integrity | Non-SELECT SQL is refused by `run_sql` — 100% of adversarial cases, asserted in unit tests + CI | 100% |
| CI regression | ~10-case subset must not regress below thresholds on every push | pass |

Scores are **measured, never estimated**. The judge model and rubric are named in the
eval report; runs without an API key produce no quality numbers.

## Scope

**In:** synthetic SQLite DB (trucks, maintenance_log, fault_codes, parts); three tools
(`run_sql` read-only, `lookup_part`, `generate_report`); single-orchestrator agent loop
(Anthropic tool calling); golden dataset (~40 cases incl. adversarial); eval harness with
xlsx results; Streamlit ops dashboard; GitHub Actions CI; Streamlit Cloud deploy.

**Out:** real telematics data, multi-agent planning, auth/multi-tenant, write operations
of any kind, fine-tuning.

## Risks & mitigations

- **Prompt-injected or adversarial SQL writes** → `run_sql` validates statements
  (single statement, SELECT-only, deny-listed keywords, row cap) and opens SQLite in
  read-only mode; asserted in tests and CI.
- **Judge flakiness / fabricated metrics** → explicit rubric, named judge model, cached
  transcripts committed for CI; missing key ⇒ metrics reported as "not measured".
- **Demo cost/latency** → small model for routine calls, row caps, cached eval runs.
- **License hygiene** → reference repos are MIT; any borrowed pattern is labeled in-code.
